# AIBrainIM P1 进展记录（2026-05-07 上午）

## 本轮完成

### 1. 首页驾驶舱再收口一层
这轮我没有去碰你已经在改的 `ProjectLibraryScreen.tsx`，避免并行开发互相踩文件，而是把 **首页 Dashboard** 再往“可用产品”方向收了一刀，让首屏更像真正的 AI 大脑驾驶舱，而不是一堆平铺信息：

- 新增 **「首页重点」spotlight 区**，把首页最该顶到前面的三件事固定成首屏重点卡片：
  - AI 产出现在处于什么状态
  - 还有多少项待人工拍板
  - P1 当前是不是已经从样板走向可用驾驶舱
- 这三张卡都直接可点跳转，分别连到：
  - 调度链
  - 需确认项
  - 项目库
- 文案上也进一步去掉工程自嗨味，改成用户能直接判断当前局面的表达，不再只是“系统里有什么数据”，而是“现在最值得看什么”。

这一步的核心意义很直接：**首页从“信息展示页”继续往“决策驾驶舱”收口，AI 产出流、调度状态、待确认项、项目推进这四条主线开始更清楚地争夺首屏优先级。**

### 2. Dashboard 对应测试补上
为了不只是改 UI，我顺手把这轮首页收口补了独立测试：

- 新增 `__tests__/DashboardScreen.test.tsx`
- 覆盖内容包括：
  - 新增的「首页重点」区是否渲染出来
  - “P1 正在从样板走向可用驾驶舱” 是否存在
  - 点首页重点卡是否能正确跳到 `ProjectLibrary`
  - “看调度链” 快捷动作是否仍能正常跳转 `DispatchChain`

这样这轮不是单纯改视觉，而是把首页信息收口行为也纳入测试保护，后面继续改 Dashboard 时不容易把主链路悄悄改坏。

### 3. 本轮执行策略
这轮有一个明确工程判断：**仓库当前已有未提交的 `src/screens/ProjectLibraryScreen.tsx` 修改和对应新测试文件，我没有去覆盖它，而是选择改 Dashboard 这一侧的邻接模块。**

这样做的好处是：
- 避免把上一轮或并行中的项目库修改踩掉
- 继续推进 P1 可用闭环，不因为局部文件占用而停住
- 让首页与项目库形成前后呼应：首页先聚焦“现在要看什么”，项目库承接“这些项目具体怎么展开”

### 4. 本轮验证目标已准备好
本轮改动已经把测试文件补齐，下一步就是直接跑：
- `npm test -- --runInBand`
- 如有需要再补 `npm run typecheck`

当前还没看到结构性阻塞，仍然属于可以直接连续推进的状态。

## 还差什么

### 1. 真实 live 会话还没做一轮移动端实测
代码层的协议映射已经又补实了一层，但还没在真实 Gateway 会话上跑一轮：
- 消息发送
- session 回流
- 任务映射
- 首页/任务页/智能体页同步显示

### 2. Chat / Dispatch 的真实字段还可以再压一层
现在 `tasks` 这层更稳了，但 `dispatches` 还主要依赖本地回填和运行时补丁，后面还需要继续把真实会话字段往 dispatch 视图压。

### 3. TestFlight 链路还是差最后一段 Apple 侧配置
工程和前台都在收口，但 Apple Developer / App Store Connect / 截图这几项外部物料还没真正落地。

## 下一步

1. 继续补 `dispatch` 这一层的真实协议字段映射，让首页 AI 产出流和调度链更少 fallback 味。
2. 用真实 Gateway 配置跑一轮移动端 live 闭环验证，确认任务映射层在真数据下成立。
3. 然后再推进 TestFlight 上线链路的 Apple 侧收口。
