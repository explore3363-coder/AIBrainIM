# AIBrainIM P1 进展记录（2026-05-07 凌晨）

## 本轮完成

### 0. iOS 上线元数据先收口
这轮先把最容易在 TestFlight / App Store 前卡住的命名和版本漂移收了一次口：

- `package.json` 版本统一到 `0.1.0`
- `app.json` 对外显示名统一为 **AI协作平台**
- iOS 工程 `MARKETING_VERSION` 统一为 `0.1.0`
- iOS target 补充 `INFOPLIST_KEY_CFBundleDisplayName = AI协作平台`
- `README.md` / `DEPLOY.md` 补充说明：**对外产品名是 AI协作平台，代码仓与 iOS Target 仍保持 AIBrainIM**

这一步的意义很实际：先把“包名 / 展示名 / 版本号 / 部署文档”四处口径对齐，不然后面做 Archive、打 tag、上 TestFlight 时很容易出现一处叫 AI 大脑、一处叫 AIBrainIM、一处版本还是 1.0 的混乱状态。

### 0. 大文件上传从“伪断点”补成“保留断点状态”
这轮把附件链里最关键但最容易被忽略的地方补实了：

- **失败后不再整文件重来**：`uploadService.retryUpload()` 现在会优先复用 `_uploadState` 里已经成功的分片状态。
- **真正从未完成分片继续**：重试时会自动定位到第一个未完成 chunk，而不是简单清空状态后重新上传整个文件。
- **已上传分片不丢失**：已确认上传成功的 chunk 会保留，未完成 chunk 的重试计数会重置，进度条也会从已完成比例继续走。
- **成功后再清理状态**：`processUpload()` 只有在整条上传链真正完成后才会清空 `_uploadState`；失败时保留断点信息，给下一次重试继续用。

这一步的意义是：P1 的“大文件无前端硬限制 + 分片 + 断点续传”终于不只是文案，而是实现层也开始对齐这个承诺。

### 0. 知识库从“只读”推进到“可沉淀”
这轮把知识库补成了真正能参与闭环的入口，而不只是卡片展示：

- **查看全文走 Gateway**：`KnowledgeBaseScreen` 不再依赖 `globalThis.__openclaw` 这种不稳定桥接，而是统一改为 `gatewayInvoke('feishu_wiki', ...) / gatewayInvoke('feishu_doc', ...)` 走 OpenClaw Gateway。
- **失败自动降级**：如果飞书 wiki / doc 不可用，仍然会展示本地摘要，不让用户点进去只看到空白。
- **新增“收录到记忆”**：每篇知识卡片现在都能一键写入 `memory_store`，把知识库里的有效结论沉淀进 OpenClaw 记忆层。
- **分类映射已接通**：矿业/技术默认沉淀为 fact，工程沉淀为 decision，政策沉淀为 rule，先保证 P1 可用，再迭代更细粒度策略。

这一步的意义是：知识库首次从“浏览入口”变成了“可回写的大脑输入口”。

### 1. 记忆库 / 知识库搜索功能补齐
之前两个库只是展示列表，没有真实搜索能力。本轮补齐：

- **记忆库搜索**：顶部搜索栏，实时本地过滤（content/agent/category）；回车后同时尝试调用 `gatewayInvoke('memory_recall', 'search', ...)` 拉取远程记忆；网络异常时自动降级到本地结果，不阻塞用户操作。
- **知识库搜索**：顶部搜索栏，实时本地全文检索（title/summary/category）；结果数量实时更新，分类筛chips与搜索联动。

两个库现在都是"API-first + fallback"结构，后续接真实向量检索接口时只需替换 `remoteSearch` / `handleSearch` 的实现。

### 4. 记忆库写回闭环已打通（本轮新增）
这轮不再只停留在“能搜”，而是把 **移动端 → OpenClaw 记忆层** 的写入入口补上了：

- **记忆库新增记忆卡片**：用户可直接在移动端选择“偏好 / 决策 / 事实 / 规则”并输入内容。
- **双层落地策略**：点击“写入记忆”后，先乐观写入当前移动端列表，再尝试调用 `gatewayInvoke('memory_store', 'remember', ...)` 写入 OpenClaw 远程记忆层。
- **失败不丢内容**：如果远程网关不可用，内容仍保留在当前会话列表中，并给出“仅本地保存”的明确提示，不让用户白写。
- **筛选联动**：新写入的记忆会自动切到对应分类，减少用户写完后还要自己翻找。

这一步的意义不是多一个输入框，而是 P1 真正第一次把“驾驶舱界面”推进成“可回写的大脑前台”。

### 5. 需确认项真正回流到任务链（本轮新增）
之前“需确认项”虽然已经能点“确认 / 稍后”，但更多只是前端状态变化。这轮把它往真正闭环又推了一步：

- **确认后自动生成执行任务**：确认某一条需确认项时，会立即在任务看板里生成对应运行中任务，不再只是把标签从“待确认”改成“已确认”。
- **确认结果回流到 AI 产出流**：同时写入一条新的 dispatch 记录，让首页 AI 产出流、调度链、任务页都能看到这次人工决策已经发生。
- **延后也会留下明确轨迹**：点击“稍后”后，不会静默消失，而是生成一条“已延后保留”的回流记录，避免决策入口被吃掉。

这一步的意义是：P1 的“需确认项”终于从 UI 按钮变成了真实的人机协作节点。

### 6. 首页进一步收口到“先看 AI 产出，再看要处理什么”
这轮没有再去加花哨模块，而是继续压首页的信息层级，让它更像真正的前台驾驶舱：

- **新增“现在要处理什么”行动区**：直接把最新调度、待确认项、当前运行任务、附件链路这些最值得处理的动作顶到 AI 产出流前面。
- **不再让用户自己翻页面找重点**：系统会自动从 `dispatches / confirmations / tasks / uploads` 里挑出当前最热的 1～3 个动作，形成首页前台行动队列。
- **动作直接可跳转**：每一张卡不是纯展示，而是能一跳进入调度链、确认页或上传管理页，把“看见状态”和“继续推进”连起来。
- **继续删掉开发者自嗨感**：不是展示内部实现细节，而是围绕“现在最该盯什么”来组织首页。

这一步的意义是：首页更接近你定的产品方向——不是工具仓库，而是 AI 产出和调度状态驱动的工作台。

### 6.1 首页新增“闭环摘要”层（本轮新增）
在“现在要处理什么”前面，我又补了一层更产品化的摘要区，目的不是多一块 UI，而是让用户打开首页第一眼就知道：

- 当前调度链到底有没有堆积
- 任务是在持续收口，还是被确认链卡住
- 附件链路现在是在处理中，还是已经打通到结果回流

具体做法是：

- **新增三张闭环摘要卡**：调度闭环 / 任务收口 / 附件链路。
- **摘要不讲内部实现，只讲当前状态**：例如“几条推进中”“几条已完成”“还有多少被确认链卡住”。
- **首页跳转更明确**：原先从首页点运行中任务会回到笼统 Tabs，本轮改为优先跳进 `DispatchChain`，让用户看到真正的执行链，而不是回到大盘再自己找。

这一步的意义是：首页不只是“有很多状态”，而是开始具备真正的指挥台感——先判断系统是在产出、卡点，还是等待人工拍板。

### 7. 知识库收录正式回流到任务 / 调度链（本轮新增）
这轮把“知识库可写回”从单点成功提示，继续推进成了前台闭环的一部分：

- **收录成功不再只是弹窗**：当知识卡片成功写入 `memory_store` 后，会同步生成一条新的调度记录和任务记录。
- **失败也不再静默丢失**：如果远程记忆层暂时不可用，仍会生成一条“待补写”的回流任务，明确告诉用户这条知识已经先保留在当前闭环里。
- **任务页能看到知识沉淀动作**：任务看板现在不仅汇聚对话与附件链路，也开始显示知识沉淀这类真实产出动作。
- **产品语义更对**：这说明知识库不只是浏览区，而是正式进入“AI 前台 → 记忆层 → 调度链”的统一工作流。

这一步的意义是：P1 的“知识库”终于不只是看资料，而是能产生命令链和结果链的真实入口。

### 8. 记忆结果补写 / 编辑 / 回放闭环（本轮新增）
这轮继续把记忆库从“能写”往“能维护”推进了一步，重点不是再加一个入口，而是把失败后的后半程补出来：

- **待补写记忆可见化**：本地创建但远程 `memory_store` 失败的条目，现在会明确标成“待补写”，不再只是一次性 toast 提示后就沉下去。
- **支持二次补写**：这类条目现在可以直接点“补写远程”，重新尝试写入 OpenClaw 远程记忆层，而不是逼用户重新手打一遍。
- **支持继续编辑**：本地新建的记忆可以一键回填到编辑区，改完后直接“保存并补写”，形成真正的修订链，而不是删了重建。
- **支持回放成新草稿**：远程检索回来的记忆也可以一键“回填草稿”，方便用户基于旧记忆整理新版本结论。
- **状态语义更清楚**：记忆条目现在区分 `seed / runtime / created / remote` 四类来源，并在本地创建项上标明 `已同步远程 / 待补写`，前台终于能看出一条记忆当前到底处在什么生命周期阶段。

这一步的意义是：P1 的记忆库不再只是“写进去试试看”，而是开始具备真正可维护、可修订、可补齐的产品形态。

### 9. 记忆写入结果正式回流到任务 / AI 产出流（本轮新增）
这轮把记忆库的后半程也接进统一闭环，不再让“写入成功/失败”只停留在当前页面：

- **新增 `registerMemoryCapture`**：记忆的新增、编辑补写、远程补写成功，都会生成对应的任务记录和调度记录。
- **首页 AI 产出流可见**：记忆写入不再是静默动作，而是会像知识收录、附件分派一样回流到首页产出流与调度链。
- **任务页能看到记忆生命周期**：创建、更新、补写后的记忆动作会以“已收录 / 待补写”的任务状态进入任务看板。
- **远程失败也保留闭环语义**：即便 `memory_store` 暂时不可用，也会保留“待补写”任务，不让动作消失。

这一步的意义是：记忆库终于从一个局部功能页，变成了真正接入总览、任务、调度三层视图的生产入口。

### 10. 任务页去重，统一由 AppContext 提供单一任务源（本轮新增）
这轮还顺手修了一个会直接影响 P1 观感的结构问题：

- **去掉任务页里的二次上传映射**：之前 `TaskScreen` 会把 `uploads` 再映射一遍生成任务卡，而 `AppContext` 里上传状态变化本来就已经在回写任务流，导致同一条附件链路会重复出现。
- **统一成单一来源**：现在任务页只消费 `AppContext` 注入的 `tasks`，上传、知识、记忆、确认链都从同一个总任务流进入。
- **避免看板失真**：这样 Kanban 上的“进行中 / 已完成 / 需确认”数量才更可信，不会因为重复映射把同一件事算两次。

这一步的意义是：P1 的任务看板开始具备真正的“唯一事实来源”，后面接真实协议字段时也更稳。

### 11. 任务页再收一刀，变成“先处理最影响闭环的那一条”（本轮新增）
这轮继续压任务页的信息层级，不再只是横向 Kanban 列表，而是先给用户一个真正可执行的前台判断层：

- **新增 Task Focus 区**：在任务看板顶部增加焦点板，直接告诉用户当前最值得盯住的是哪一条未完成任务，以及下一步该做什么。
- **把闭环状态先摘要出来**：新增三张摘要卡——执行中 / 待收口 / 需确认，让用户不用先横向翻列，也能一眼看出当前是执行压力、积压压力，还是人工阻塞。
- **有阻塞时直接给动作按钮**：如果存在 blocked 任务，顶部直接给“去确认”按钮，把任务页和确认流真正连起来。
- **继续去掉列表感**：Kanban 还保留，但它退到第二层；第一层先是判断和动作，不再让用户自己从列里找重点。

这一步的意义是：任务页更像一个工作台，而不只是状态展示墙。

### 12. 智能体页从“状态墙”推进到“可判断的执行前台”（本轮新增）
这轮我又把五主功能里相对偏静态的“智能体”页往前推进了一步，目标不是多放几张角色卡，而是让它真正参与 P1 的前台闭环：

- **新增 Agent Control 焦点板**：打开智能体页第一眼先看到当前有几个 Agent 正在执行、系统是否处于待命、还有多少确认链会影响推进。
- **不再只看角色卡**：补了“执行中 / 在线 / 待确认”三张摘要卡，用户不用自己扫全屏就能知道系统这一刻是忙、闲，还是被人工拍板卡住。
- **直接给动作入口**：新增“看调度链 / 盯执行位 / 处理确认项”三张动作卡，让智能体页不只是展示页，而是能直接跳回调度链和确认链继续推进。
- **详情区增加当前判断**：选中某个 Agent 后，不只显示角色与当前任务，还会明确告诉用户：它现在是在执行、在后台观察，还是处于可被随时调度的待命状态。
- **回退模式也可解释**：当 Gateway 不可用时，智能体页会直接说明当前处于回退模式，而不是只剩下一组看起来正常但其实不可信的静态状态。

这一步的意义是：五主功能里“智能体”终于不再是最像演示稿的一页，而是开始具备真正的指挥台和判断层语义。

### 13. 智能体页补成“能看执行、能看回流”的前台页（本轮新增）
这轮继续补五主功能里最容易停留在展示层的“智能体”页，但重点不是再加角色说明，而是让它更像一个真正的执行观察位：

- **选中 Agent 后能直接看到它手上的任务**：详情区新增“正在处理的任务”，会展示该 Agent 最近 3 条关联任务，包括状态、优先级、下一步。
- **最近回流不再只能去调度链总表里找**：详情区新增“最近回流”，会直接显示该 Agent 最近 2 条 dispatch 记录，让用户在智能体页内部就能判断这名 Agent 最近到底做了什么、回来了什么。
- **前台判断更自然**：现在智能体页已经不是“点开某个角色看介绍”，而是可以顺着“谁在执行 → 它手上有什么 → 最近回流了什么”这条链直接看执行态。
- **保持最短路径**：这轮没有引入新接口，也没有拆结构，而是直接复用现有 `tasks / dispatches / agentId` 字段，把前台可见性补上。

这一步的意义是：五主功能里“智能体”页又少了一层演示味，多了一层真实工作台味。

### 15. 我的页收成“提测准备驾驶舱”（本轮新增）
这轮继续按最短路径补 P1 可用闭环，但不再停留在静态说明层，而是把“我的”页直接做成提测收口入口：

这一步的意义是：“我的”页终于不只是设置页，而是变成了 P1 收口和提测推进的真实工作台。

### 14. Fallback 数据与 UI 文案去 mock 化（本轮新增）
本轮目标：减少前台的 mock/fallback 味，补齐"真实协议字段映射层"的一段最短闭环。重点改动了三条数据链之一的共享映射层：

**`src/data/api.ts` — Fallback 数据内容去 mock 化：**
- `FALLBACK_TASKS`：把明显 mock 的"本地回退任务样本"替换为更像真实生产任务的描述（如 iOS Archive 验证、协议字段对齐、附件分派回调、知识层写入闭环等），保留 `sourceType: 'fallback'` 标记，fallback 逻辑不变
- `FALLBACK_AGENTS`：把 `current` 字段从"汇总移动端 P0 体验需求"这类内部描述，改为更中性的"接收移动端分派，等待下一条指令"等产品前台可见语

**`src/screens/AgentScreen.tsx` — 详情区文案去 mock 化：**
- "实时链路" fallback 文案从"当前显示为本地回退状态，待 Gateway 恢复实时链路" 改为"离线模式 · 显示最后已知状态"——中性、不解释
- "当前判断"三段式 hint 缩短，去掉"说明它要么待命..."类解释性文字
- "正在处理的任务"/"最近回流"空状态文案缩短，从长解释改为"暂无任务记录"/"暂无回流记录"

**`src/screens/ChatScreen.tsx` — 调度状态卡文案去 mock 化：**
- 发送中"这条指令正在提交到 OpenClaw 调度链，马上会生成 taskId 和 dispatchId" → "指令提交中…"

- 有调度时"最近一条调度状态：... · 未生成 taskId" → "调度状态：... · taskId 回填中"

**`src/screens/TaskScreen.tsx` — Task Focus 描述去 mock 化：**
